# User Roles

## Overview

LiturgyBridge serves different types of users with different responsibilities.

The platform separates:

- people who consume information
- people who contribute content
- people who manage communities
- people who maintain the platform

Permissions should follow responsibilities.

---

# 1. Visitor

## Description

A visitor can use LiturgyBridge without becoming a registered member.

Typical example:

A person enters a church, scans a QR code, and follows the service.

## Permissions

Can:

- view public liturgical content
- select language
- follow the current service
- read translations
- access public community information

Cannot:

- access private community areas
- upload content
- manage groups

---

# 2. Registered User

## Description

A person with a LiturgyBridge account.

A registered user can participate in communities.

## Permissions

Can:

- join communities
- manage personal settings
- select preferred languages
- receive notifications
- save preferences

---

# 3. Community Member

## Description

A registered user who belongs to a parish or community.

Examples:

- parish member
- regular visitor
- volunteer

## Permissions

Can:

- access member-only information
- view internal calendar events
- receive community notifications
- participate in approved groups

---

# 4. Group Member

## Description

A member of a specific community group.

Examples:

- choir member
- youth group member
- volunteer team

## Permissions

Can:

- access group resources
- view group schedules
- participate in group communication

Examples:

Choir:

- hymn texts
- sheet music
- rehearsal dates

---

# 5. Content Contributor

## Description

A person allowed to add or update specific content.

Examples:

- translator
- choir coordinator
- community editor

## Permissions

Can:

- create documents
- upload resources
- suggest corrections
- maintain assigned content

Content changes may require approval.

---

# 6. Community Administrator

## Description

A trusted person responsible for managing a community space.

Examples:

- parish administrator
- parish secretary
- designated volunteer

## Permissions

Can:

- create community events
- manage members
- manage groups
- send notifications
- manage community resources

Cannot:

- change global platform settings

---

# 7. Priest / Clergy

## Description

A clergy role with responsibilities related to worship services.

## Permissions

Can:

- prepare upcoming services
- select liturgical templates
- approve liturgical content
- manage service-related information

The priest role does not automatically imply technical administration rights.

---

# 8. Translator / Liturgical Editor

## Description

A specialist contributing translations and liturgical resources.

## Permissions

Can:

- create translations
- review language content
- maintain terminology
- contribute to liturgical libraries

Changes should support review workflows.

---

# 9. Platform Administrator

## Description

Maintains the technical platform.

## Permissions

Can:

- manage system configuration
- maintain infrastructure
- handle technical support

Should not automatically have access to private community content.

---

# Permission Principles

## Least Privilege

Users receive only the permissions required for their responsibility.

---

## Separation of Roles

Technical administration and community authority should remain separate.

---

## Community Ownership

Each community controls:

- its members
- its groups
- its internal content

---

## Future Role Extensions

Possible future roles:

- monastery administrator
- diocese administrator
- choir director
- educator
- researcher
- archivist
